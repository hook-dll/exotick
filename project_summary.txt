================================================================================
exotick — Developer / Maintainer Reference
================================================================================
Last updated: 2026-07-13

Everything a maintainer needs to reconstruct or modify the system. Dry and
authoritative. If it isn't here, it isn't a stable feature.


================================================================================
1. STACK
================================================================================
Frontend  React 18, TypeScript, Vite 8, Tailwind 3, React Router v6,
          react-markdown + remark-gfm
Backend   Node.js 24, Express 4, TypeScript, tsx (dev watch),
          node:sqlite (built-in — no native deps), multer, fflate (pure-JS
          zip), pdfkit, node:crypto (scrypt, sha256, randomBytes),
          cookie-parser
Database  SQLite at data/cms.db (WAL journal, foreign_keys ON)
Ports     dev: backend 3001, client 5173; prod: 3001 only (Express serves
          the built SPA)

                                      / CI path) → create admin from env
                                    - else if stdin is a TTY → prompt for
                                      username + confirmed password
                                      (askLine / askSecret via raw mode)

                                  Idempotent; safe to call every boot.
                                  Also exports resetAdmin() + ResetError

        reset-admin-cli.ts        Standalone entry for admin recovery —
                                  `npm run reset-admin` (local) or, in
                                  Docker, `docker compose exec exotick node
                                  server/dist/reset-admin-cli.js`. Runs
                                  resetAdmin(): picks the sole admin (or
                                  EXOTICK_RESET_USERNAME), sets a new
                                  password (prompt or EXOTICK_RESET_PASSWORD),
                                  clears disabled_at, revokes that admin's
                                  sessions, logs a password_reset event.
                                  Host-only; there is NO web-facing reset.
        rateLimit.ts              progressive lockout used by /login (keyed
                                  per-IP) and /change-password (keyed
                                  per-user):
                                  5 failed attempts → 5 min lock, next 5
                                  → 50 min, next 5 → 500 min (indefinite
                                  cap). Tier persists across the wait;
                                  a successful hit clears the bucket.
                                  API: check(key) / recordFailure(key) /
                                  reset(key). Idle buckets pruned every
                                  15 min (24 h without activity, only when
                                  not currently locked).

                                  library_id), PATCH /reorder (requires
                                  library_id), PATCH /bulk-move (requires
                                  library_id), POST /bulk-delete,
                                  POST /bulk-duplicate (duplicates land
                                  in the SAME library as each source),
                                  POST /bulk-copy (copies selected cases +
                                  their sections to ANOTHER library —
                                  body: target_library_id, case_ids[],
                                  section_ids[]; non-destructive, source
                                  untouched. Each source section is
                                  re-created in the target by NAME + COLOR
                                  (reused if an exact name+color match

                                  the source color); copied cases + notes

        testRuns.ts               GET /?status=&library_id= (both

                                  GET /:id (adds items[]; items strip
                                  updated_at for the same privacy
                                  reason — snapshots + status + updated_by
                                  only). POST / requireCanRun — compose;
                                  derives library_id from selected
                                  case_ids and REJECTS with 400 if the
                                  selection spans multiple libraries;
                                  also rejects runner_name that resolves
                                  to an admin. POST /:id/start
                                  requireCanRun + assertOwnership;
                                  POST /:id/take-over requireCanRun —
                                  reason ≥ 10 chars, cooldown window
                                  read from settings.take_over_cooldown_
                                  minutes (0 disables the check; 10-char
                                  reason still applies), 409 with a vague
                                  "This runner seems to be active" body
                                  (no minutes-left leaked). ONLY active
                                  runs are eligible — drafts (composing)
                                  and completed runs are rejected with

                                  success.
                                  PATCH /items/:itemId requireCanRun +
                                  assertOwnership + stamps updated_by;
                                  POST /:id/finish requireCanRun +
                                  assertOwnership + stamps updated_by on

                                  requireRole('editor') (editors: DRAFT
                                  runs only; ACTIVE + COMPLETED → admin)
        auth.ts                   GET /me (public — returns { user } or

3. DATABASE SCHEMA
================================================================================

libraries
  id             INTEGER PK AUTOINCREMENT
  name           TEXT NOT NULL
  order_index    INTEGER NOT NULL DEFAULT 0
  created_at     DATETIME DEFAULT CURRENT_TIMESTAMP

sections
  id             INTEGER PK AUTOINCREMENT
  name           TEXT NOT NULL
  order_index    INTEGER NOT NULL DEFAULT 0
  color          TEXT                          NULL, or one of red/orange/
                                                yellow/green/blue/purple
  library_id     INTEGER NOT NULL REFERENCES libraries(id) ON DELETE CASCADE
  created_at     DATETIME DEFAULT CURRENT_TIMESTAMP

test_cases
  id             INTEGER PK AUTOINCREMENT
  section_id     INTEGER REFERENCES sections(id) ON DELETE SET NULL
  description    TEXT NOT NULL                 title/name of the case
  notes          TEXT                          markdown; may embed /uploads/*
  order_index    INTEGER NOT NULL DEFAULT 0
  library_id     INTEGER NOT NULL REFERENCES libraries(id) ON DELETE CASCADE
  created_at     DATETIME DEFAULT CURRENT_TIMESTAMP

test_runs
  id             INTEGER PK AUTOINCREMENT
  name           TEXT NOT NULL
  runner_name    TEXT                          current runner (may change on
                                                take over); NULL only in
                                                degenerate cases
  status         TEXT NOT NULL DEFAULT 'composing'
                 CHECK(status IN ('composing','active','completed'))
  library_id     INTEGER NOT NULL REFERENCES libraries(id) ON DELETE RESTRICT
                                                  — the FK backstops the
                                                  app-layer 409 guard when
                                                  a library has runs
  created_at     DATETIME DEFAULT CURRENT_TIMESTAMP
  started_at     DATETIME
  finished_at    DATETIME

test_run_items
  id                       INTEGER PK AUTOINCREMENT
  test_run_id              INTEGER NOT NULL REFERENCES test_runs(id) ON DELETE CASCADE
  test_case_id             INTEGER REFERENCES test_cases(id) ON DELETE SET NULL
  snapshot_description     TEXT NOT NULL       copied at compose time
  snapshot_notes           TEXT                copied at compose time
  snapshot_section_name    TEXT                copied at compose time
  order_index              INTEGER NOT NULL DEFAULT 0
  status                   TEXT CHECK(status IN ('pass','fail','skip'))
                                                NULL until marked
  updated_at               DATETIME            written on every mark/finish
  updated_by               TEXT                username of the actor;
                                                NULL only on rows written
                                                before the column existed

settings                   key/value store. Known keys:
                             app_name                       branding name
                             app_logo_file                  branding logo
                             take_over_cooldown_minutes     admin-tunable
                                                            take over
                                                            cooldown;
                                                            default 60,
                                                            range 0-10080.
                             (edit_password_hash rows from previous versions
                             are ignored — password mode is gone.)
  key                      TEXT PK
  value                    TEXT

event_log                  Append-only log of named actions:
                              login           — successful authentication
                              edit            — any mutation to libraries /
                                                sections / test cases
                              compose         — new test run created
                              start           — run entered active state
                              finish          — run entered completed state
                              take_over       — runner_name changed via
                                                the take over flow (also
                                                carries reason +
                                                previous_runner)
                              password_change — user changed their own
                                                password
                              password_reset  — admin reset someone else's
                                                password; target username
                                                stored in the reason field
                                                (`target: <username>`)
                            Rows persist independently of test_runs so
                            deleting a run does NOT wipe log history
                            (test_run_id → NULL, test_run_name stays).
                            Deliberately NOT a full audit trail: no
                            reads, no individual case marks, no
                            actor-role snapshot. NOT included in backup
                            exports. An earlier version was named
                            `audit_log`; db.ts migrates it in place
                            if legacy rows are present.
  id                       INTEGER PK AUTOINCREMENT
  created_at               DATETIME DEFAULT CURRENT_TIMESTAMP
  event_type               TEXT NOT NULL       'take_over' initially
  actor_username           TEXT NOT NULL
  test_run_id              INTEGER REFERENCES test_runs(id) ON DELETE SET NULL
  test_run_name            TEXT                snapshot — survives run
                                                deletion so the row stays
                                                readable
  previous_runner          TEXT                who owned the run before
  reason                   TEXT                the ≥10-char reason typed
                                                at take over time

users
  id             INTEGER PK AUTOINCREMENT
  username       TEXT NOT NULL UNIQUE COLLATE NOCASE
  password_hash  TEXT NOT NULL                 scrypt$N$r$p$saltB64$hashB64
  role           TEXT NOT NULL
                 CHECK(role IN ('admin','editor','runner','watcher'))
  created_at     DATETIME DEFAULT CURRENT_TIMESTAMP
  disabled_at    DATETIME                      NULL = active; soft-disable
  last_login_at  DATETIME

sessions
  id             TEXT PK                       sha256(rawSid) base64url —
                                                the raw ID lives only in
                                                the cookie
  user_id        INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE
  created_at     DATETIME DEFAULT CURRENT_TIMESTAMP
  expires_at     DATETIME NOT NULL             30-day sliding window
  user_agent     TEXT
  ip             TEXT

Indexes:
  idx_sessions_user_id         ON sessions(user_id)
  idx_sessions_expires_at      ON sessions(expires_at)
  idx_sections_library_id      ON sections(library_id)
  idx_test_cases_library_id    ON test_cases(library_id)
  idx_test_runs_library_id     ON test_runs(library_id)
  idx_event_log_created_at     ON event_log(created_at)
  idx_event_log_test_run_id    ON event_log(test_run_id)

Migrations (all idempotent, run on server start):
  ALTER TABLE test_cases        ADD COLUMN notes TEXT
  ALTER TABLE test_run_items    ADD COLUMN snapshot_notes TEXT
  ALTER TABLE test_run_items    ADD COLUMN updated_by TEXT
  ALTER TABLE sections          ADD COLUMN color TEXT
  Rebuild users table when the CHECK constraint's SQL doesn't contain
  'watcher' (detected via sqlite_master.sql):
    CREATE TABLE users_new ...; INSERT SELECT; DROP; RENAME;
  Libraries migration (runs once per legacy install):
    1. CREATE TABLE libraries (…)
    2. INSERT "Main" row if libraries is empty (id = <DEFAULT_LIBRARY_ID>)
    3. ALTER TABLE sections   ADD COLUMN library_id INTEGER NOT NULL
                              DEFAULT <DEFAULT_LIBRARY_ID> REFERENCES

    4. ALTER TABLE test_cases ADD COLUMN library_id INTEGER NOT NULL
                              DEFAULT <DEFAULT_LIBRARY_ID> REFERENCES

  ALTER TABLE ADD COLUMN with NOT NULL DEFAULT populates every existing
  row with the default value, so all legacy sections/cases/runs land in

4. AUTH MODEL
================================================================================

Every /api request (other than /api/auth/login, /api/auth/logout,
/api/auth/me, /api/health, and /api/branding GET) requires a valid session
cookie. requireAuth resolves the cookie against sessions.id (hashed) and
attaches req.user + req.sessionId. requireRole compares req.user.role
against the endpoint's minimum via the strict rank {watcher:1, runner:2,
editor:3, admin:4}.

On first launch (no admin in the DB) the server prompts interactively for
an admin username + password when stdin is a TTY. In headless / Docker
contexts the credentials come from EXOTICK_ADMIN_USERNAME +
EXOTICK_ADMIN_PASSWORD instead. Without either path, the server refuses
to start. Bootstrap is idempotent — subsequent boots are a no-op.

Password recovery: a locked-out admin resets from the host via the
reset-admin CLI (see §10/§12); non-admins ask an admin to reset them at
Settings > Users. There is no email / web-facing self-reset by design.

Roles — hierarchy is admin > editor > runner > watcher for the read /
edit / management chain, BUT admin is deliberately excluded from the
runner workflow. See requireCanRun in server/src/auth/middleware.ts.

  Capability                            admin  editor  runner  watcher
  ------------------------------------  -----  ------  ------  -------
  Manage users                          ✅     ❌      ❌      ❌
  Change branding (name + logo)         ✅     ❌      ❌      ❌
  Backup export/import                  ✅     ❌      ❌      ❌
  Edit sections / test cases            ✅     ✅      ❌      ❌
  Delete DRAFT (composing) test runs    ✅     ✅      ❌      ❌
  Delete ACTIVE / COMPLETED test runs   ✅     ❌      ❌      ❌
  Upload images (notes)                 ✅     ✅      ❌      ❌
  Compose a run                         ❌     ✅      ✅      ❌
  Start / mark / finish a run           ❌     ✅ *    ✅ *    ❌
  Take over a run                       ❌     ✅      ✅      ❌
  View everything                       ✅     ✅      ✅      ✅

  * — Only the current runner of THAT run can start/mark/finish. Any
      editor or runner can take over (subject to reason + cooldown).
      POST /test-runs also rejects runner_name that resolves to an admin.

The single admin is bootstrapped from env vars and does not appear in the
Users management UI. The UI never offers 'admin' in role pickers; roles
are chosen among editor/runner/watcher for all created users.


================================================================================
5. SESSION MODEL
================================================================================

Cookie:              exotick_sid (HttpOnly, SameSite=Lax, Secure when
                     req.secure via trust proxy=1)
Cookie value:        32 random bytes, base64url (43 chars) — the raw ID
DB storage:          sha256(rawSid) base64url as sessions.id — a leaked
                     sessions table cannot be used to forge cookies
Lifetime:            30 days sliding — expiry rewrites when > 24 h since
                     the last slide

Rate limit on /api/auth/login: progressive per-IP lockout, in-memory.
  Tier 1: 5 failed attempts     → lock 5 min
  Tier 2: next 5 failed attempts → lock 50 min
  Tier 3: next 5 failed attempts → lock 500 min (stays here indefinitely)
Wrong-user and wrong-password fail identically (no enumeration surface).
Returns 429 with a Retry-After header (seconds) and a retryAfterMs field
in the JSON body. Any successful login on the same IP clears everything.
State is process-local — a server restart resets all lockouts, which is
usually what you want during recovery.

Rate limit on /api/auth/change-password: SAME ladder, but keyed
per-user instead of per-IP (key = `pwchange:${user_id}`). A stolen
session that starts guessing the current password locks the change-
password endpoint for the whole account, across all of that user's
sessions. Malformed input (new password < 8 chars) does NOT count as a
failed attempt against the current password. Successful change clears
the bucket.

Password change / disable / reset kill all sessions for that user
except the caller's own where relevant.


================================================================================
6. TEST RUN OWNERSHIP + TAKE OVER
================================================================================

Ownership check (assertOwnership in routes/testRuns.ts):
  Fires on POST /:id/start, PATCH /items/:itemId, POST /:id/finish.
  Loads the run row; compares run.runner_name to req.user.username. Non-
  owners get 403 with a specific error mentioning the current runner.

Take over (POST /:id/take-over):
  Requirements:
    authenticated req.user.username (implicit — every /api call is authed)
    reason.trim().length ≥ 10 chars (400 otherwise)
    run.status === 'active' — drafts and completed runs are BOTH
      rejected with 400 "Only active runs can be taken over."
      Composing (draft) is deliberately blocked so take over can't be
      used to steal someone else's in-progress compose.
    cooldown since last activity (settings.take_over_cooldown_minutes,
      default 60, admin-configurable; 0 disables timing, reason still
      required):
      Activity is MAX(items.updated_at) or, absent that, run.started_at.
      Comparison runs entirely in SQL via
      strftime('%s','now') - strftime('%s', last_activity) to avoid the
      SQLite-UTC-without-Z / JS-parses-as-local footgun.
      409 with a vague body — no minutes-left leaked on the wire.
  Effect: UPDATE test_runs SET runner_name = <caller> WHERE id = ?
          + writeEvent('take_over', ..., previous_runner, reason)
  Returns updated TestRun.

Per-item authorship:
  updated_by is written on every PATCH /items and on auto-skips during
  POST /:id/finish.
  Contributors summary (History Detail chips + PDF section) aggregates
  distinct updated_by values with counts, sorted desc.

Composed runner vs current runner:
  runner_name is BOTH — take over rewrites it. The event_log records who
  took what over (with reason) — see §? and routes/log.ts. The distinct
  actors in items.updated_by additionally tell you
  who actually did work.

  npm run dev
  → tsx watch server/src/index.ts (backend 3001)
  → vite (client 5173 with proxy: /api /uploads /branding → 3001)

  npm run build
  → vite build (client → client/dist)
  → tsc      (server → server/dist)

  npm start
  → cross-env NODE_ENV=production node server/dist/index.js
  Express serves the SPA at /* and 404s /api/*-not-found as JSON.

Docker:
  docker compose up -d --build
  Multi-stage Dockerfile: node:24-alpine builder installs all deps and
  runs the build; runtime stage copies server/dist + client/dist +
  prod-only node_modules into a slim image, non-root exotick user.
  HEALTHCHECK: wget /api/health every 30 s.
  Compose mounts a named volume exotick-data at /data. Container binds
  to 127.0.0.1:3001; Caddy on the host terminates TLS.

HTTPS:
  The app speaks plain HTTP on :3001. TLS termination is out of scope —
  operators put a reverse proxy in front (Caddy, nginx, Traefik) and get
  certs however they choose. Recommended path in readme.md is a real
  domain + Let's Encrypt via Caddy; deploy/Caddyfile.example is a small
  starter for that, but nothing in the repo mandates any specific proxy.

Graceful shutdown:
  SIGTERM / SIGINT close the HTTP server with a 10s force-exit timer.
  Ensures Docker rolling restarts don't chop in-flight backup exports.

11. NON-OBVIOUS BEHAVIORS / GOTCHAS
================================================================================

- SQLite CURRENT_TIMESTAMP writes UTC as 'YYYY-MM-DD HH:MM:SS' WITHOUT
  a timezone marker. `new Date(str)` parses that as LOCAL time on V8
  (Node + Chromium). Any elapsed-time computation on the client must
  route through util/serverDate.ts::parseServerTs, which normalizes to
  'YYYY-MM-DDTHH:MM:SSZ' first. On the server, take over cooldown uses
  SQL strftime('%s', 'now') - strftime('%s', last_activity) to avoid the
  issue entirely. Pure display code (`.toLocaleString()` on runs / items
  / sessions) does NOT use parseServerTs and can appear offset by the
  operator's timezone; acceptable but a known cosmetic bug.

- Take over cooldown is admin-tunable via Settings > Take over cooldown
  (persisted in settings.take_over_cooldown_minutes). Server is
  authoritative on both the threshold and whether it's currently in
  effect: routes/testRuns.ts returns cooldown_active as a boolean per run
  (computed in SQL from strftime diffs). The client never receives raw
  last-activity timestamps for runs it doesn't own, so it can't work out
  when the runner last acted. Setting cooldown to 0 disables the timing
  check entirely; the 10-character reason still applies.

- Session cookie value ≠ sessions.id in DB. Cookie carries the RAW id;
  DB stores sha256(raw). resolveSession hashes the incoming cookie
  before lookup. destroySessionByRawId (used in logout) hashes;
  destroySessionByStoredId (used in the revoke-by-URL endpoint) does not.
  Row IDs returned by listSessionsForUser ARE the stored (hashed) ids —
  the client uses them opaquely for the DELETE /sessions/:id endpoint.

- Bootstrap admin is idempotent: no-op when any active admin exists.
  Consequently, an attacker with shell access to the DB can delete the
  admin row and then re-run the bootstrap to create a new admin with a
  chosen password. Mitigation deliberately not implemented; app-level
  defense doesn't win against host-shell access, and preventing it would
  break legitimate recovery. That same host access is the intended
  recovery channel for a locked-out admin — the reset-admin CLI (§10/§12)
  is the clean version of it (no DB surgery, revokes the admin's sessions).

- Backup zip (routes/backup.ts) covers only the test-case library and
  uploaded images referenced by notes. It does NOT include:
    - run history (test_runs, test_run_items)
    - users table
    - sessions table
    - branding folder (data/branding/*)
  Backup import accepts both 'exotick-backup' and legacy 'testcms-backup'
  format strings. Replace mode deletes only test_cases then sections;
  test_run_items survive via ON DELETE SET NULL.
  Import guardrails: 100 MB uploaded, 2000 entries, 25 MB per entry, 500 MB
  total inflated — beyond that the archive is rejected.

- POST /api/samples/load ALWAYS creates a new library called "Samples"
  (or "Samples (2)" if that name is taken). Never modifies existing
  libraries.

- BrandingProvider fetches /api/branding at boot; document.title is
  synced via useEffect. Custom logo lives in data/branding/, NOT
  data/uploads/. Backup export won't include it.

- Vite dev proxy must forward /branding as well as /api /uploads.
  Missing entry produces broken logo images in dev (production is
  single-origin so it works there).

- The Compose page's runner field is a <select> populated by
  /api/users/roster. Watchers are ALWAYS filtered out (they can't be
  runners); admins are filtered out for non-admin callers.

- cooldown status on GET /api/test-runs is a server-computed
  cooldown_active boolean per row (SQL CASE derived from
  strftime diffs against settings.take_over_cooldown_minutes). The
  raw last_activity_at timestamp is never returned — that would leak
  when the current runner last acted. Client uses the boolean
  directly to enable/disable the Take over button; no local math.

- The admin's own row is filtered out of the Users table in Settings
  (client-side). The admin manages their own password via the Change
  my password card. Prevents accidental self-disable / self-delete.

- Roles: 'admin' is bootstrap-only. UI role pickers (create + per-row
  dropdown) only offer editor/runner/watcher. Server VALID_ROLES accepts
  'admin' for completeness but there's no path to create one from the UI.

- No CORS middleware — the app is single-origin (Vite dev proxy,
  same-origin Caddy in prod). Adding CORS would require also revisiting
  the SameSite=Lax cookie posture.

- Uploads reject SVG explicitly. Both file extension AND multipart
  Content-Type must match a raster-image whitelist (PNG/JPG/GIF/WebP for
  notes, PNG/JPG/WebP for branding). This closes stored XSS via SVG
  <script>.

- /uploads and /branding are served with X-Content-Type-Options: nosniff
  as defense in depth so a browser can't override Content-Type.

- trust proxy is set to 1 (not `true`). With a single Caddy in front, this
  correctly resolves req.ip and honors X-Forwarded-Proto. With no proxy,
  clients can no longer spoof X-Forwarded-For to fool the rate limiter.

12. QUICK RECOVERY OPS
================================================================================

Reset the admin password (locked-out admin — preferred):
  npm run reset-admin                         # local; prompts for new pw
  docker compose exec exotick node server/dist/reset-admin-cli.js   # Docker
  (or set EXOTICK_RESET_PASSWORD for an unattended reset). Revokes the
  admin's sessions; no DB surgery needed. See readme > Password recovery.

Reset a non-admin user's password:
  Admin does it in the UI at Settings > Users > Reset password (also
  revokes that user's sessions).

Reset all users (retain library + runs):
  node -e "const {DatabaseSync}=require('node:sqlite');
  const db=new DatabaseSync('data/cms.db');
  db.exec(\"DELETE FROM users; DELETE FROM sessions;\");"

Then restart with bootstrap env vars.

Wipe everything (empty install):
  Delete data/cms.db, data/cms.db-shm, data/cms.db-wal, and (optionally)
  data/uploads/, data/branding/.

Inspect current state:
  node -e "const {DatabaseSync}=require('node:sqlite');
  const db=new DatabaseSync('data/cms.db');
  console.log(db.prepare('SELECT id,username,role,disabled_at FROM users').all());"

Force-clear rate limit (survives process only):
  restart the server.
